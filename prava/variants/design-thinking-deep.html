Design Process Case Study

From Technical Requirements to Human-Centered Solution

1
Define

Problem Discovery

The brief: Implement file sharing for email client with "proper" access control. Technical requirements specified A, R, W, M, D permissions (Administer, Read, Write, Manage, Delete).

Initial Specification

Technical diagram showing permission matrix. Focus on system capabilities, not user needs.

The Real Problem

Users don't think: "I need to set Write and Read permissions for user@domain.com"

Users actually think: "I need to share this with Anna so she can edit it"

Key Insight: The requirement asked for "rights management" but users needed "people sharing". We were solving the wrong problem.
2
Explore

Iteration & Sketching

Paper prototypes explored different approaches. Each iteration tested: How can we shift focus from permissions to people?

Iteration A

Card Grid (3×3)
Visual selection of people. ON/OFF toggles. "Recommended" section. Explored: Can cards make selection obvious?

Iteration B

Column Layout
Name | Toggle | Share. Highlighted selection states. Explored: Does structure add clarity?

Iteration C

Contact List
Simplified names (JAN, FRANTA). Prominent search. Explored: List vs. card format trade-offs.

Testing Hypotheses

  • Can visual cards make people-selection more intuitive than checkboxes?
  • Does "recommended contacts" reduce search time?
  • Should selection state be shown via highlighting or separate list?
  • How do we balance quick access (cards) with large contact lists (search)?
3
Refine

Design Principles

Through iterations, clear principles emerged that guided the final design:

Match Mental Models

Users think in terms of people ("Anna", "Petr"), not technical IDs or permission codes.

Reduce Cognitive Load

Show what matters: people's names and faces. Hide technical details until needed.

Progressive Disclosure

Simple defaults cover 80% of cases. Advanced options available via modal.

Search-First

For large contact lists, search is fastest. Cards provide quick access to recent/recommended.

Visual Feedback

Pills show selections. Highlights show state. No ambiguity about what's selected.

Human Language

Translate technical jargon: "Read/Write" → "Who can edit?"

4
Deliver

Final Solution

The working prototype combines all learned principles: search, cards, pills, human language, progressive disclosure.

Interactive prototype: Type → Select → Share

What We Removed

  • Technical permission codes (A, R, W, M, D)
  • Access control matrices and checkboxes
  • The concept of "rights management"
  • Everything between user intent and action

What We Added

  • Search with autocomplete
  • Visual cards for recommended contacts
  • Pills showing current selection
  • Human-language permissions ("Who can edit?")
  • Smart defaults (most common case)

Finding what to remove is more valuable than deciding what to add.

This case study demonstrates design thinking in practice: understanding the real problem, exploring solutions through iteration, refining based on principles, and delivering a human-centered solution.
